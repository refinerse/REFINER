# REFINER: Task-Focused and Self-Validation Agent for Resolving Code Review Comments

REFINER is a code refinement agent that resolves pull-request reviewer comments via two mechanisms:

1. **Task-focused resolution** — classifies each comment into a task type (bug-fixing, refactoring, documentation, other) and extracts the reviewer's *why*, *concern*, and *expectation* to guide the agent.
2. **Self-validation** — synthesizes an executable validation test that fails on the pre-refined code and passes once the concern is addressed, giving the agent a concrete pass/fail signal during refinement.

Evaluated on the [c-CRAB](https://github.com/c-CRAB-Benchmark/dataset) benchmark (483 review comments, 339 PRs, 76 open-source repositories) across three backbone models: **Qwen 3.6 Flash**, **Qwen 3.6 Plus**, and **Claude Sonnet 4.6**.

## Dataset

| File | Description |
|------|-------------|
| `dataset/instances.jsonl` | All 339 PR instances with comments, diff hunks, Docker image references, and ground-truth assessment tests |
| `dataset/comment_task_gt.jsonl` | Human-verified task-type labels for all 483 comments (Bugfix / Refactoring / Documentation / Others) |

Each line in `instances.jsonl` is one PR instance. Key fields:

- `instance_id` — unique identifier (e.g. `sympy__sympy-26515@2c01ca9`)
- `repo` — GitHub repository slug
- `commit_to_review.head_commit` — the commit the agent is checked out to
- `comments` — list of review comments with `text`, `path`, `diff_hunk`
- `assessment_tests` — ground-truth test code per comment

## Finding Prompts, Trajectories, and Diffs

Every instance folder under `agents_results/<method>/<instance_id>/` contains:

| File | Description |
|------|-------------|
| `prompt.txt` | The exact prompt sent to the coding agent |
| `trajectory.json` | Full agent trajectory — tool calls, LLM outputs, test results, git diffs at each step |
| `agent.diff` | Final `git diff` produced by the agent |
| `result.json` | Resolution outcome per comment (`resolved`, `test_passed`, `test_output`) |
| `qwen_stdout.txt` | Raw stdout from the Qwen Code CLI |
| `qwen_stderr.txt` | Raw stderr from the Qwen Code CLI |
| `intent_map.json` | *(REFINER variants only)* Task type + why/concern/expectation per comment |
| `validation_tests.diff` | *(REFINER with self-validation only)* Generated validation test files as a diff |

**Example** — inspect the REFINER prompt and trajectory for one instance:

```bash
cat agents_results/pure_qwen_3.6_plus_with_task_and_validation_test/sympy__sympy-26515@2c01ca9/prompt.txt
cat agents_results/pure_qwen_3.6_plus_with_task_and_validation_test/sympy__sympy-26515@2c01ca9/intent_map.json
python traj_to_md.py agents_results/pure_qwen_3.6_plus_with_task_and_validation_test/sympy__sympy-26515@2c01ca9/trajectory.json
```

**Validation test generation** results live at `agents_results/results_testgen_<model>/<instance_id>/`:

| File | Description |
|------|-------------|
| `result.json` | Test generation outcome (success, whether test fails on pre-refined, passes under GT patch) |
| `test_review_comment_N.py` | Generated test file for comment index N |

**Task classification** outputs live in `task_classification/`:

| File | Description |
|------|-------------|
| `comment_task_qwen.jsonl` | Task type + why/concern/expectation for each comment, classified by Qwen 3.6 Plus |
| `comment_task_qwen36flash.jsonl` | Same, classified by Qwen 3.6 Flash |
| `comment_task_claude.jsonl` | Same, classified by Claude sonnet 4.6|

Each JSONL line has fields: `instance_id`, `comment_index`, `label` (task type), `why`, `concern`, `expectation`.

## Per-Assertion Patch Verification (`evaluate_agent_patch_on_split/`)

The `evaluate_agent_patch_on_split/` directory mirrors the method names in `agents_results/` and stores results from running split acceptance tests (one sub-test per assertion). Each instance folder contains `result.json` with per-assertion pass/fail details.

## Environment

All experiments were conducted on a Linux machine with **32 CPU cores** and **125 GB RAM**. Each PR instance runs inside its own Docker container; parallel workers (`--workers N`) each occupy one container, so available RAM is the main constraint on concurrency.

## Installation

**1. Python environment**

Requires Python ≥ 3.12. Install dependencies with [uv](https://github.com/astral-sh/uv):

```bash
pip install uv      # or: curl -Lsf https://astral.sh/uv/install.sh | sh
uv sync
```

This installs `litellm`, `datasets`, and `pytest` as declared in `pyproject.toml`.

**2. Docker**

Docker must be installed and running. Pull the c-CRAB benchmark images listed in `dataset/instance_images.csv`:

```bash
while IFS=, read -r instance_id image_name; do
    docker pull "$image_name"
done < dataset/instance_images.csv
```

**3. Qwen Code CLI** (required for Qwen-based runs)

Install and authenticate the Qwen Code CLI, then verify the settings file exists at `~/.qwen/settings.json`.

**4. API keys** (required for Claude-based runs)

```bash
export ANTHROPIC_API_KEY=<your-key>
```

## Reproducing Experiments

### Prerequisites

- Docker running with c-CRAB images pulled (see Installation above)
- `uv sync` completed
- Qwen Code CLI authenticated (`~/.qwen/settings.json`) for Qwen runs
- `ANTHROPIC_API_KEY` set for Claude runs

### Step 1 — Classify comment tasks (task-focused resolution)

```bash
# Classify with Qwen 3.6 Plus
python classify_comment_task.py \
    --dataset-file dataset/instances.jsonl \
    --output task_classification/comment_task_qwen.jsonl \
    --model qwen-plus
```

### Step 2 — Generate validation tests (self-validation)

```bash
python run_batch_testgen.py \
    --dataset-file dataset/instances.jsonl \
    --output-dir agents_results/results_testgen_qwen_3.6_plus \
    --model qwen-plus
```

### Step 3 — Run the refinement agent

**Vanilla agent:**
```bash
python run_batch_agent_resolution.py \
    --dataset-file dataset/instances.jsonl \
    --output-dir agents_results/pure_qwen_3.6_plus \
    --model qwen-plus
```

**REFINER (task-focused + self-validation — full system):**
```bash
python run_batch_agent_resolution_vt_sk.py \
    --dataset-file dataset/instances.jsonl \
    --output-dir agents_results/pure_qwen_3.6_plus_with_task_and_validation_test \
    --intent-file task_classification/comment_task_qwen.jsonl \
    --testgen-dir agents_results/results_testgen_qwen_3.6_plus \
    --model qwen-plus
```

**Ablation — task-focused only:**
```bash
python run_batch_agent_resolution_with_task.py \
    --dataset-file dataset/instances.jsonl \
    --output-dir agents_results/pure_qwen_3.6_plus_with_task \
    --intent-file task_classification/comment_task_qwen.jsonl \
    --model qwen-plus
```

**Ablation — self-validation only:**
```bash
python run_batch_agent_resolution_validation_test.py \
    --dataset-file dataset/instances.jsonl \
    --output-dir agents_results/pure_qwen_3.6_plus_with_validation_test \
    --testgen-dir agents_results/results_testgen_qwen_3.6_plus \
    --model qwen-plus
```

Use `--workers N` to run N instances in parallel. Use `--repo <slug>` to process a single repository. Results accumulate incrementally; re-running skips already-completed instances unless `--no-resume` is passed.

### Step 4 — Generate comparison tables and metrics

```bash
# Main comparison (RQ1, RQ2, RQ3)
python compare_validation_test_vs_pure_qwen_replay.py

# Per-assertion split evaluation
python report_eval_on_split.py
```

## Key Scripts Reference

| Script | Purpose |
|--------|---------|
| `classify_comment_task.py` | Classify review comments into task types (task-focused resolution, Step 1) |
| `run_batch_testgen.py` | Generate validation tests for all instances (Step 2) |
| `run_batch_agent_resolution.py` | Vanilla agent — no augmentation |
| `run_batch_agent_resolution_with_task.py` | Agent + task-focused resolution only |
| `run_batch_agent_resolution_validation_test.py` | Agent + self-validation only |
| `run_batch_agent_resolution_vt_sk.py` | REFINER — task-focused + self-validation (full system) |
| `compare_validation_test_vs_pure_qwen_replay.py` | Produce all summary tables and resolution metrics |
| `report_eval_on_split.py` | Per-assertion split evaluation metrics |
| `traj_to_md.py` | Convert a `trajectory.json` to a readable Markdown report |
| `patch_similarity.py` | Compute function-level BLEU and edit similarity vs. ground-truth patch |
| `analyze_task_vt_effect.py` | Analyze the downstream effect of task classification and VT quality on resolution rate (RQ3.2) |

## Pipeline Modules (`pipeline/`)

| Module | Description |
|--------|-------------|
| `agent_resolver.py` | Vanilla Qwen Code resolver — baseline prompt and lifecycle |
| `agent_resolver_with_task.py` | Intent-augmented resolver — adds task type + why/concern/expectation to prompt |
| `agent_resolver_validation_test.py` | Validation-test resolver — injects generated test as a pass/fail signal |
| `agent_resolver_vt_sk.py` | REFINER resolver — combines task-focused resolution and self-validation |
| `test_generator.py` | Validation test generation loop (generate → execute on pre-refined → feedback) |
| `llm_client.py` | Unified LLM client (Qwen / Claude) |
| `dataset_utils.py` | Dataset loading and comment matching helpers |
