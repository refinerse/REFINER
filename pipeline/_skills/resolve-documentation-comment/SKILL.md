# Resolve Documentation Comment

## Task overview

You are given a code review comment on a PR. Your job is to apply the requested documentation change to the repository mounted at `/workspace`.

**Inputs:**

| Field | What it contains |
|---|---|
| `review_comment` | The reviewer's request — may be terse or indirect |
| `diff_hunk` | The `@@ ... @@` context showing the file state at the time of review |
| `file` | The file path relative to the repo root |

---

## Workflow

### Step 1 — Interpret the review comment

Documentation review comments are often terse. Common patterns and what they mean:

**Inline `suggestion` block**
The reviewer has written the exact replacement text. Apply it verbatim.

**"Typo" or points out a wrong word/phrase**
Find and replace the exact incorrect string. The reviewer often quotes or highlights it.

**"This link is wrong / no longer accurate"**
Update the URL or anchor. Infer the correct target from the surrounding context, the diff `+` lines, or the reviewer's description.

**"Update / fix the docstring"**
The code behaviour changed and the docs don't reflect it. Rewrite the relevant section to accurately describe the new behaviour shown in the diff.

**"Add field X" (e.g. `version_added`, issue references)**
Insert the missing field in the right place using the format established by surrounding entries.

**"Remove X from docs"**
Delete the line, option, or phrase. Verify the surrounding content still reads correctly.

**"This description is misleading / doesn't make sense"**
Rewrite to accurately describe what the code actually does. Use the diff and surrounding code as the source of truth.

**"Use X markup, not Y" (e.g. `C(...)` not `O(...)`)**
Replace every instance of the incorrect markup with the correct one in the relevant block.

**"X isn't autogenerated" / "remove this marker"**
Delete the incorrect comment or marker.

**"No spaces in names" / "make link clickable" / style nits**
Apply the exact style correction the reviewer describes.

---

### Step 2 — Use the diff hunk for location and context

The `-` lines show the current state in `/workspace`. The `+` lines show the author's attempted change — they are a strong hint toward the target wording, but the reviewer's comment may refine them further.

Use the hunk to:
- Find the exact location in the file
- Understand what text exists now
- Get a sense of the intended replacement shape

---

### Step 3 — Read the target file

Open the file and read the region around the diff hunk location. Understand:
- The file format (Python docstring, Markdown, RST, YAML, etc.)
- The indentation and markup conventions used nearby
- The full sentence or block that needs changing (not just the single line in the hunk)

---

### Step 4 — Make the edit

Apply the smallest change that satisfies the reviewer's request. The edit stays within the documentation layer — do not change logic or behaviour.

**File format notes:**
- **Python docstrings** — preserve indentation; for raw strings (`r"""`), keep the `r` prefix
- **Markdown / MDX** — preserve heading levels, list style, and link format `[text](url)`
- **RST / changelogs** — use correct roles (`:func:`, `:class:`, `:issue:`) and maintain list alignment
- **YAML doc blocks** — preserve indentation level; insert fields in the order established by neighbouring entries
- **Inline `#` comments** — replace only the comment text, leave the code line untouched

---

### Step 5 — Self-check

1. Does the edited text accurately reflect what the reviewer asked for?
2. Is the old incorrect text completely gone?
3. If the reviewer gave verbatim replacement text, does my edit match it exactly?
4. Did I change only what was needed — no unrelated reformatting?
5. Does the surrounding text still read correctly after the edit?

---

## Key pitfalls

- **Reviewer gives the problem, not always the exact wording.** If there's no `suggestion` block, infer the correct text from the diff `+` lines and the reviewer's description — then check both agree.
- **When a `suggestion` block and the diff `+` lines differ**, the `suggestion` block takes priority.
- **Multi-comment instances.** A single instance can have several review comments on different files. Apply each independently.
- **Minimal edits only.** Reformatting unrelated lines can cause unexpected failures.
- **Preserve format-specific conventions.** Incorrect indentation or missing markup (e.g. forgetting `r"""`) can break rendering or tests.
